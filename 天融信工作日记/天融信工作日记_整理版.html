3.1 标准镜像构建流程

将一个 Golang 项目打包成镜像并在 Kubernetes 上运行，主要分为两大步骤：使用 Docker 将 Go 应用打包成镜像，然后在 K8s 集群中部署。

编写 Dockerfile（多阶段构建）

# 第一阶段：构建阶段
FROM golang:1.19-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

# 第二阶段：运行阶段（极简镜像）
FROM scratch
COPY --from=builder /app/main .
EXPOSE 8080
CMD ["./main"]
关键点 多阶段构建使最终镜像极小。CGO_ENABLED=0 编译纯静态二进制。scratch 是最小基础镜像，需要 Shell 可换用 alpine（约 5MB）。

构建与推送

docker build -t your-username/your-go-app:latest .
docker login
docker push your-username/your-go-app:latest

编写 K8s 配置文件

Deployment (deployment.yaml) 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: your-go-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: your-go-app
  template:
    metadata:
      labels:
        app: your-go-app
    spec:
      containers:
      - name: your-go-app
        image: your-username/your-go-app:latest
        ports:
        - containerPort: 8080
        resources:
          requests: { memory: "64Mi", cpu: "250m" }
          limits:   { memory: "128Mi", cpu: "500m" }
        livenessProbe:
          httpGet: { path: /healthz, port: 8080 }
          initialDelaySeconds: 3
          periodSeconds: 10
        readinessProbe:
          httpGet: { path: /ready, port: 8080 }
          initialDelaySeconds: 5
          periodSeconds: 10
Service (service.yaml) 
apiVersion: v1
kind: Service
metadata:
  name: your-go-app-service
spec:
  selector:
    app: your-go-app
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
  type: LoadBalancer

部署到集群

kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
kubectl get deployments
kubectl get pods
kubectl get services

3.2 镜像上传与运行

在本机打包完成镜像后，有几种方法将镜像放到客户机上替换服务。

方法 1：Docker Save/Load（推荐，离线环境首选）

# 步骤1：保存镜像为 tar 文件
docker save -o cleansvc-new.tar cleansvc:latest

# 步骤2：传输到客户机
scp cleansvc-new.tar user@customer-machine:/tmp/

# 步骤3：在客户机加载镜像
docker load -i /tmp/cleansvc-new.tar
docker images | grep cleansvc

# 步骤4：替换 K8s 服务
kubectl rollout restart deployment cleansvc -n sdsp
# 或直接更新镜像
kubectl set image deployment/cleansvc cleansvc=cleansvc:latest -n sdsp

方法 2：使用私有 Docker Registry

docker login internal-registry.company.com
docker tag cleansvc:latest internal-registry.company.com/sdsp/cleansvc:latest
docker push internal-registry.company.com/sdsp/cleansvc:latest
kubectl set image deployment/cleansvc cleansvc=internal-registry.company.com/sdsp/cleansvc:latest -n sdsp

验证部署

kubectl rollout status deployment/cleansvc -n sdsp
kubectl get pods -n sdsp -l app=cleansvc
kubectl logs -f deployment/cleansvc -n sdsp

3.4 离线环境编译部署

Makefile 可用部分

buildi:
	@echo "Start building docker image......"
ifneq ($(ARCH),sw64)
	@go mod tidy
	@go mod vendor
endif
	@docker build \
		--build-arg BUILDER_IMAGE=$(builderImage) \
		--build-arg BASIC_IMAGE=$(basicImage) \
		--build-arg BUILD_CGO=$(CGO_ENABLED) \
		--build-arg BUILD_ARCH=$(ARCH) \
		--build-arg LDFLAGS=$(ldFlags) \
		-t $(targetImage):$(VERSION) -f ./docker/Dockerfile ./
	@docker tag $(targetImage):$(VERSION) $(targetImage):latest

离线环境操作步骤

# 1. 设置环境变量
export ARCH=amd64

4.6 ClickHouse Too many parts

来源：NGDLP/clickhouse/现场问题.md + NGDLP/clickhouse/基本操作.md（合并 ClickHouse 操作与问题）
错误信息 Too many parts (300). Merges are processing significantly slower than inserts

核心原因：数据插入频率太高，每次插入数据量太小，导致 Part 生成速度远大于后台 Merge 线程处理速度，Part 数量超过默认阈值（300）。

紧急处理

# 1. 停止或大幅降低写入频率（5-10 分钟让后台 Merge 线程处理）

# 2. 查看当前表活跃 Parts 数量
SELECT count() FROM system.parts WHERE table = 'sensitive_file' AND active = 1;

# 3. 手动触发合并（异步操作，建议业务低峰期执行）
OPTIMIZE TABLE sensitive_file FINAL;

# 4. 临时放宽阈值（治标不治本，谨慎使用）
SET max_parts_in_total = 500;

可能遇到的错误

OPTIMIZE TABLE 报错：CANNOT_STATVFS Could not calculate available disk space (statvfs), errno: 2, strerror: No such file or directory
原因：ClickHouse 数据目录或临时目录的 PVC 未正确挂载。解决：kubectl exec -it -n topsec-topngdlp statefulset.apps/clickhouse -- df -h 检查挂载情况。

根本原因排查

  • 单条插入：循环执行 INSERT INTO table VALUES，每次生成新 Part
  • 表引擎：非 MergeTree 引擎（Log、Memory、Buffer）没有 Merge 机制
  • 系统资源：CPU 打满或磁盘 I/O 延迟过高导致 Merge 变慢
  • 并发数：background_pool_size 配置过小

5.3 goms resource 配置解析

来源：项目问题/.../resource解析.md

goms 的 resource 配置使用 YAML 格式定义处理器，通过 Bloblang 脚本指定数据路由（meta catalog）和转换规则（root）。每个资源类型对应一个独立配置文件。

配置文件结构

processor_resources:
    - label: "processor_resource_sensitive_category_level"
      bloblang: |
       meta catalog   = "sensitive_category_level"
       root           = this
       root.message_type = deleted()

字段说明

字段说明
label处理器唯一标识，命名规范 processor_resource_{资源类型}
bloblang数据转换和路由脚本，使用 Bloblang 语言
meta catalog设置元数据，标识资源类型
root消息主体赋值

四种资源类型配置

Go 代码 key配置文件meta catalog
filesprocessor_resource_sensitive_file.yamlsensitive_file
categoriesprocessor_resource_sensitive_file_category.yamlsensitive_file_category
levelsprocessor_resource_sensitive_file_level.yamlsensitive_file_level
category_levelsprocessor_resource_sensitive_category_level.yamlsensitive_category_level

gRPC 流式调用的日志拦截器，记录 STREAM-START 和 STREAM-END 的耗时与错误信息。

func StreamLoggingInterceptor(srv interface{}, ss grpc.ServerStream,
    info *grpc.StreamServerInfo, handler grpc.StreamHandler) error {
    startTime := time.Now()
    ctx := ss.Context()
    var addr string
    if client, ok := peer.FromContext(ctx); ok {
        addr = client.Addr.String()
    }
    logx.WithContext(ctx).Infof("[RPC] [STREAM-START] %s - %s", addr, info.FullMethod)
    err := handler(srv, ss)
    duration := time.Since(startTime)
    if err != nil {
        logx.WithContext(ctx).Errorf("[RPC] [STREAM-END] %s - %s, duration: %v, error: %v",
            addr, info.FullMethod, duration, err)
    } else {
        logx.WithContext(ctx).Infof("[RPC] [STREAM-END] %s - %s, duration: %v",
            addr, info.FullMethod, duration)
    }
    return err
}
