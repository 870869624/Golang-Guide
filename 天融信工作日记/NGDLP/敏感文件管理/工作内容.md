## 一、模块概述

负责NGDLP产品中敏感文件的**全生命周期管理**：上传、下载、提取、列表查询、导出、处置。

涉及的技术组件：
- gRPC（Agent文件上传）
- MongoDB（文件元数据存储、状态管理）
- ClickHouse / LogMiner（多维度聚合查询）
- NATS（异步事件通知）
- Redis（缓存）

## 二、敏感文件上传（gRPC流式）

### 核心文件
- `upload_file_sensitive_logic.go` - gRPC流式上传主逻辑
- `filechunk/` - 分片管理、加密合并、解密

### 业务流程

```
Agent ──gRPC stream──> 服务端
  │
  ├── 1. 循环Recv接收分片，写入Buffer
  ├── 2. 收到EOF，结束接收
  ├── 3. 参数校验（filename, md5, level, category, rule, update_type, tenant_id）
  ├── 4. 创建临时目录 /tmp/ngdlp/sensitive_file/{md5_hash}/{timestamp}
  ├── 5. 写入分片（chunk.Write）
  │
  └── 如果是最后一片（isFinal=true）：
      ├── 6. 获取文件级锁（sync.Map + Mutex）
      ├── 7. 合并分片 + AES-GCM加密 → 写入临时文件 {filepath}.tmp.{nanotime}
      ├── 8. os.Rename 临时文件 → 正式文件路径（原子写入）
      ├── 9. 释放文件锁
      └── 10. 提交到MongoDB批量写入队列（FileBatchWriter / AutoUploadFileBatchWriter）
```

### 关键设计决策

#### 1. 并发写入控制 - sync.Map文件锁

**问题**：多个Agent可能同时上传同一个文件（相同路径），导致并发写入冲突。

**方案对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全局Mutex | 最简单 | 粒度太大，不同文件之间也互斥 |
| map[string]*Mutex + RWMutex | 细粒度 | 需要额外加锁保护map本身 |
| **sync.Map** | **内置并发安全，LoadOrStore原子操作** | **不适合读多写少以外场景** |

**最终选择**：sync.Map，因为：
- 文件路径作为key是动态的（不同文件不同路径）
- LoadOrStore 原子性保证"不存在就创建锁，存在就复用"
- 代码更简洁

```go
var fileLocks sync.Map

lockKey := data.FilePath
mu, _ := fileLocks.LoadOrStore(lockKey, &sync.Mutex{})
fileMu := mu.(*sync.Mutex)
fileMu.Lock()
// ... 写入文件 ...
fileMu.Unlock()
```

#### 2. 原子写入 - 临时文件 + Rename

**问题**：如果合并分片+加密过程中断（进程崩溃、磁盘满），会留下损坏的文件。

**方案**：
1. 先写入临时文件 `{filepath}.tmp.{nanotime}`
2. 合并+加密全部完成后，`os.Rename` 为正式路径
3. Rename 在同一文件系统上是原子操作
4. 如果中途失败，defer会清理临时文件

```go
tempFilePath := data.FilePath + ".tmp." + strconv.FormatInt(time.Now().UnixNano(), 10)
// ... 合并加密写入tempFilePath ...
if err = os.Rename(tempFilePath, data.FilePath); err != nil {
    // 原子写入失败
}
```

#### 3. 分片校验机制

- 每片有 `chunkHash` 校验
- 整文件有 `fileHash` 和 `MD5` 三重校验
- 校验在 `chunk.MergeEncrypt` 内部完成

#### 4. 文件存储路径设计

```
{SensitiveFilePath}/{tenantID}/{md5}{.ext}       # 本地平台
{SensitiveFilePath}/{platform_hash}/{tenantID}/{md5}{.ext}  # 非本地平台
```

- 按租户隔离（tenantID）
- 文件名用MD5+原始扩展名，天然去重（相同MD5只存一份）
- 平台维度通过platform hash隔离

#### 5. 异步写入MongoDB

文件上传成功后，不直接写MongoDB，而是提交到批量写入队列：
- `FileBatchWriter` - 手动上传的文件
- `AutoUploadFileBatchWriter` - 自动上传的文件

队列内部做批量合并，减少MongoDB写入次数。

## 三、敏感文件下载

### 核心文件
- `download_sensitive_file_logic.go`

### 业务流程

```
1. 根据fileID + tenantID + platform 查找文件记录
2. 如果没找到，尝试按MD5 + tenantID + extract_state=成功 模糊匹配
3. 找到后，创建临时解密目录 /tmp/ngdlp/sensitive_file/{tenantID}
4. 调用 chunk.MergeDecrypt 解密文件
5. 如果解密失败（gcm open failed），尝试用fileID作为密钥重试（兼容多版本密钥）
6. 返回解密后的文件路径和文件名
```

### 关键设计

#### 接口抽象（便于测试）

```go
type downloadSensitiveFileFinder interface {
    FindByFileID(ctx context.Context, fileID, tenantID, platform string) (*sensitive.File, error)
    FindByFilter(ctx context.Context, filter bson.M, opts ...*options.FindOptions) ([]sensitive.File, error)
}
```

通过接口解耦数据访问层，可以mock实现进行单元测试。

#### 多版本密钥兼容

下载时先尝试用MD5作为密钥解密，如果失败（gcm open failed），再尝试用fileID作为密钥。这是为了兼容加密密钥版本变更的场景。

## 四、敏感文件提取

### 核心文件
- `extract_logic.go` - 提取主逻辑
- `extract_logic_v1.go` - V1版本（供内部调用）

### 业务流程

```
1. 前端选择敏感文件 → 调用Extract接口
2. 根据文件ID列表，通过LogMiner/ClickHouse查询文件详情
3. 向Agent下发提取任务（TaskSendNotCreateTask）
   - taskType: sensitive_file_extract
   - execType: immediately（立即执行）
   - 包含文件路径、MD5等信息
4. 异步插入MongoDB提取记录，状态设为"提取中"
5. 查找相同MD5的其他文件记录，批量插入无状态提取记录
```

### 关键设计

#### MD5去重关联

同一个文件（相同MD5）可能在多个终端上被发现。提取时：
1. 先提取用户选中的文件
2. 再根据这些文件的MD5，查找所有相同MD5的其他文件
3. 过滤掉已选中的，对剩余的批量插入"无状态提取记录"（没有提取状态追踪）

#### 批量插入优化

```go
const batchSize = 50
for i := 0; i < len(bulkData); i += batchSize {
    end := i + batchSize
    if end > len(bulkData) {
        end = len(bulkData)
    }
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    err := l.svcCtx.Model.SensitiveFile.BulkInsertSkipExisting(ctx, bulkData[i:end])
    cancel()
}
```

- 每批50条
- 每批30秒超时
- SkipExisting：已存在则跳过（幂等）

## 五、敏感文件列表查询

### 核心文件
- `file_list_logic_v1.go` - V1版本列表查询（主要使用）
- `file_list_logic.go` - 原始版本

### 业务流程

```
1. 构建查询条件（match）：
   - tenant_id（必须）
   - client_id（可选，支持逗号分隔多个）
   - 如果未指定client_id，根据用户角色查权限范围内的client_id列表
   - file_storage, file_id, update_type（精确匹配）
   - filename, suffix, sha1, md5, category, rule, rule_tag（模糊匹配/wildcard）
   - level（多值匹配）
   - platform（模糊匹配）

2. 调用LogMiner/ClickHouse查询，带聚合和排序：
   - 聚合：按文件去重统计（total_client）、UUID列表（groupUniqArray）
   - 排序：discovery_time DESC, id DESC
   - 分页：Scroll-based（基于id滚动）

3. 查询级别映射表（categoryLevelSearch），将数字级别转为可读文本

4. 关联MongoDB查询提取状态：
   - 根据文件ID列表批量查询MongoDB中的提取记录
   - 填充 existFile、extractState、extractStateEnum、extractError
```

### 关键设计

#### ClickHouse vs MongoDB 的分工

| 查询类型 | 使用组件 | 原因 |
|---------|---------|------|
| 多维度聚合统计、大范围列表 | ClickHouse/LogMiner | 列式存储，聚合快 |
| 单条记录详情、提取状态 | MongoDB | 支持事务、索引灵活 |

#### RBAC数据权限

```go
if clientIds, notFilterGroup := NewUtil(l.svcCtx).findClintIdsByRole(tenantID, roleID, userID); !notFilterGroup && len(clientIds) > 0 {
    terms["client_id"], _ = structpb.NewList(util.ToAny(clientIds))
}
```

如果用户没有指定client_id，根据角色自动过滤只能看权限范围内的终端。

#### URL编码处理

```go
if req.Filename != "" {
    decodedFilename, err := url.QueryUnescape(req.Filename)
    // ...
}
```

前端传来的文件名可能是URL编码的，需要先解码再模糊查询。

## 六、列自定义配置

### 核心文件
- `../sensitivefilecolumn/get_column_order_logic.go`
- `../sensitivefilecolumn/set_column_order_logic.go`

### 功能说明

用户可以自定义敏感文件列表的列显示顺序和显隐状态。

### 业务流程

```
1. 获取列配置（GetColumnOrder）：
   - 先查MongoDB是否有用户/租户级别的自定义配置
   - 如果没有（首次使用），返回默认配置
   - 如果有，与默认配置做diff：
     - 默认新增的列 → 自动追加（index递增）
     - 默认删除的列 → 过滤掉
   - 这样保证版本升级新增列时，老用户也能看到

2. 设置列配置（SetColumnOrder）：
   - 保存用户的列排序和显隐配置到MongoDB
```

### 默认列配置

支持三种视图的列配置：
- `client_sensitive_file` - 终端敏感文件列表（22列）
- `file_list` - 数据资产列表（18列）
- `to_be_confirmed` - 待人工确认列表（18列）
