## 一、模块概述

数据资产治理是NGDLP的核心展示模块，将终端Agent上报的敏感文件数据聚合为"数据资产"视角——同一份文件（相同MD5）在多个终端上被发现时，合并展示为一条数据资产记录，附带发现终端列表、命中规则、分类分级等信息。

## 二、数据资产列表查询（file_list_logic_v1）

### 核心文件
- `file_list_logic_v1.go`

### 查询流程

```
1. 构建ClickHouse/LogMiner查询条件：
   - tenant_id（必须）
   - client_id（可选，支持多个）
   - RBAC权限过滤（如果用户没指定client_id，根据角色查权限范围）
   - 精确匹配：file_storage, file_id, update_type
   - 模糊匹配：filename, suffix, sha1, md5, category, rule, rule_tag
   - 多值匹配：level
   - 时间范围：discovery_time
   
2. 调用LogMiner/ClickHouse查询，带聚合：
   - groupUniqArray：聚合所有发现终端的UUID
   - count：统计发现终端数
   - groupUniqArray：聚合所有client_id
   - 排序：discovery_time DESC, id DESC
   - Scroll-based分页（基于id滚动，避免深分页性能衰减）

3. 查询级别映射表（categoryLevelSearch），将数字级别转为可读文本

4. 反查MongoDB填充提取状态：
   - 根据返回的文件ID列表，批量查MongoDB提取记录
   - 填充 existFile, extractState, extractStateEnum, extractError
```

### 关键设计

#### 1. ClickHouse vs MongoDB 分工

| 查询类型 | 使用组件 | 原因 |
|---------|---------|------|
| 多维度聚合统计、大范围列表 | ClickHouse/LogMiner | 列式存储+向量化执行，聚合快 |
| 单条记录详情、提取状态 | MongoDB | 支持事务、索引灵活、实时性强 |

#### 2. Scroll分页替代Offset分页

ClickHouse深分页时，OFFSET的性能随偏移量线性下降。Scroll分页用上一页最后一条的id作为下一页的起始点：
```
第1页：WHERE id > 0  ORDER BY id ASC LIMIT 20
第2页：WHERE id > {上一页最后一条id}  ORDER BY id ASC LIMIT 20
```

#### 3. RBAC数据权限

如果用户未指定client_id，根据角色自动过滤：
```go
if clientIds, notFilterGroup := NewUtil(l.svcCtx).findClintIdsByRole(tenantID, roleID, userID); !notFilterGroup && len(clientIds) > 0 {
    terms["client_id"], _ = structpb.NewList(util.ToAny(clientIds))
}
```

#### 4. URL编码处理

前端传来的文件名可能是URL编码的，需要先解码：
```go
if req.Filename != "" {
    decodedFilename, err := url.QueryUnescape(req.Filename)
}
```

#### 5. 三个视图的统一与差异

| 视图 | 说明 | 差异 |
|------|------|------|
| `file_list` | 数据资产列表 | 按MD5去重聚合，显示发现终端数 |
| `client_sensitive_file` | 终端敏感文件列表 | 按终端维度，不聚合 |
| `to_be_confirmed` | 待人工确认列表 | 筛选需要人工确认的记录 |

## 三、列自定义配置（sensitivefilecolumn）

### 核心文件
- `get_column_order_logic.go`
- `set_column_order_logic.go`

### 功能说明

用户可以自定义敏感文件列表的列：显示/隐藏、调整顺序。

### 3.1 获取列配置（GetColumnOrder）

#### 设计思路

```
1. 先查MongoDB是否有用户/租户级别的自定义配置
2. 如果没有（首次使用） → 返回默认配置
3. 如果有 → 与默认配置做diff：
   - 默认新增的列 → 自动追加（index递增）
   - 默认删除的列 → 过滤掉
```

#### 核心逻辑：版本升级兼容

这是最关键的设计。当产品新版本新增了列字段时，老用户的自定义配置里没有这个列，需要自动补上：

```go
// 如果默认列新增了，自动追加进去
setCoiMap := make(map[string]*sensitive.ClientInfoColumnOrderItem)
for _, v := range co.Items {
    setCoiMap[v.Name] = v  // 用户已有配置
}
for _, v := range defaultCois {
    if setCoiMap[v.Name] == nil {
        maxIndex++
        co.Items = append(co.Items, &sensitive.ClientInfoColumnOrderItem{
            Name:   v.Name,
            Status: v.Status,
            Index:  maxIndex,  // 追加到末尾
        })
    }
}

// 如果默认列减少了，要过滤掉已删除的列
defaultCoiMap := make(map[string]*sensitive.ClientInfoColumnOrderItem)
for _, v := range defaultCois {
    defaultCoiMap[v.Name] = v
}
for _, item := range co.Items {
    if defaultCoiMap[item.Name] == nil {
        continue  // 跳过已删除的列
    }
    // 保留
}
```

#### 三种视图的默认配置

| 视图 | 默认列数 | 默认显示列 |
|------|---------|-----------|
| `client_sensitive_file` | 22列 | 文件名、分类、级别、发现时间、已有文件、智能分析、建议（7列） |
| `file_list` | 18列 | 文件名、文件大小、分类、级别、最早发现时间、发现终端、智能分析（7列） |
| `to_be_confirmed` | 18列 | 同file_list（7列） |

### 3.2 设置列配置（SetColumnOrder）

```go
co := &sensitive.Sensitivefilecolumnorder{
    TenantID: tenantID,
    From:     req.From,  // 哪个视图
}
for _, item := range req.Items {
    co.Items = append(co.Items, &sensitive.ClientInfoColumnOrderItem{
        Name:   item.Name,
        Status: item.Status,  // 0=隐藏, 1=显示
        Index:  item.Index,   // 排序序号
    })
}
// Upsert：有则更新，无则插入
l.svcCtx.Model.SensitiveFileColumnOrder.UpsertSensitivefilecolumnorder(l.ctx, co)
```

### 设计总结

这个模块的技术难度不高，但体现了**产品思维**：
- 考虑了版本升级时的配置兼容（新增列自动追加、删除列自动过滤）
- 三种视图使用同一套配置框架，代码复用
- 默认配置优先展示最有业务价值的列（分类、级别、智能分析）
