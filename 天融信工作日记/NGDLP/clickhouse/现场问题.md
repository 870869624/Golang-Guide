```bash
message: Too many parts (300). Merges are processing significantly slower than inserts
```

这个错误是 ClickHouse 中最经典的 “Too many parts” 问题。核心原因是：数据插入的频率太高，每次插入的数据量太小，导致数据 Part（数据块）的生成速度远大于后台 Merge（合并）线程的处理速度，最终 Part 数量超过了默认阈值（300）。

ClickHouse 为了保证高性能，要求数据在后台不断进行合并（Merge）。如果 Part 数量过多，会严重影响读写性能，甚至导致服务崩溃，因此系统会主动拒绝写入并报错。

下面帮你梳理解决步骤和预防方案：

# 一、紧急处理（立即恢复写入）

当出现此报错时，首先需要让系统恢复正常。

1. 停止或大幅降低写入频率
   这是最直接有效的方法。如果业务允许，先停止写入 5-10 分钟，让 ClickHouse 的后台 Merge 线程有时间将小的 Parts 合并成大的 Parts，直到 Parts 数量降到阈值以下。
2. 手动触发合并（如果系统负载不高）
   如果停止写入后 Parts 数量下降很慢，可以手动强制合并指定的表。


```bash
-- 查看当前表的活跃 Parts 数量
SELECT count() FROM system.parts WHERE table = 'your_table' AND active = 1;

-- 手动触发合并 (这是一个异步操作，执行后会加入合并队列)
OPTIMIZE TABLE your_table FINAL;
```

注意：OPTIMIZE FINAL 是一个比较重的操作，建议在业务低峰期或确认系统资源充裕时执行。

3. 临时放宽阈值（谨慎使用）
如果业务无法立即停止写入，且集群内存和CPU还有余量，可以临时调高限制，但这只是“治标不治本”。

```bash
-- 在 Session 级别临时调高限制 (例如调至 500)
SET max_parts_in_total = 500;
```

注意：如果 Part 堆积速度过快，单纯调高阈值可能导致 ClickHouse 内存溢出或进入“保护模式”。

# 二、根本原因排查
你需要找出为什么 Merge 速度会慢于 Insert 速度。

1. 检查是否单条插入
最常见的错误是在代码中循环执行 INSERT INTO table VALUES (1,2,3)。每执行一次 INSERT，无论数据量大小，都会生成一个新的 Part（即使是 1 条数据）。如果每秒插入 10 次，几分钟就会达到 300 个 Part。

2. 检查表引擎

- MergeTree 家族：正常的 Merge 机制。

- Distributed 表：如果是通过分布式表写入，数据会先落在分布式表的“本地缓冲区”，也可能引发此类问题。

- 非 MergeTree 引擎：某些特殊引擎（如 Log, Memory, Buffer）没有 Merge 机制，极易触发此报错。

3. 检查系统资源

- CPU：Merge 操作非常消耗 CPU。如果 CPU 已经打满（接近 100%），Merge 线程会排队，导致 Parts 堆积。

- 磁盘 I/O：如果磁盘读写延迟过高，Merge 速度会变慢。

- 并发 Merge 数：检查 background_pool_size 配置是否过小。
