## 二、RabbitMQ 连接管理

### 单例模式

```go
// lib/mq/init.go
var (
    conn    *amqp.Connection
    channel *amqp.Channel
    once    sync.Once
)

func InitMQ() {
    once.Do(func() {
        var err error
        conn, err = amqp.Dial("amqp://user:password@host:port/")
        if err != nil {
            log.Fatal("RabbitMQ连接失败:", err)
        }
        channel, err = conn.Channel()
    })
}
```

### 连接参数

| 参数 | 值 | 说明 |
|------|---|------|
| Exchange类型 | `direct` | 精确路由匹配 |
| durable | `true` | 队列/交换机持久化 |
| autoAck | `false` | 手动ACK |
| prefetchCount | `1` | 一次只处理一条（保证顺序） |

## 三、两种消费模式

### 模式1：直接订阅队列

```go
func SubscribeToQueue(queueName string, handler func(amqp.Delivery)) {
    // 1. 声明队列（durable=true）
    ch.QueueDeclare(queueName, true, false, false, false, nil)
    
    // 2. 注册消费者
    msgs, _ := ch.Consume(queueName, "", false, false, false, false, nil)
    
    // 3. 循环消费
    for msg := range msgs {
        handler(msg)
    }
}
```

### 模式2：Exchange + RoutingKey + Queue

```go
func SubscribeToExchange(exchange, routingKey, queueName string, handler func(amqp.Delivery)) {
    // 1. 声明Exchange
    ch.ExchangeDeclare(exchange, "direct", true, false, false, false, nil)
    
    // 2. 声明Queue
    ch.QueueDeclare(queueName, true, false, false, false, nil)
    
    // 3. 绑定
    ch.QueueBind(queueName, routingKey, exchange, nil)
    
    // 4. 消费
    msgs, _ := ch.Consume(queueName, "", false, false, false, false, nil)
    for msg := range msgs {
        handler(msg)
    }
}
```

### 两种模式的区别

| | 直接订阅队列 | Exchange模式 |
|---|---|---|
| 灵活性 | 固定队列 | 可通过RoutingKey路由到不同队列 |
| 扩展性 | 一个队列一个消费者 | 一个Exchange可绑多个队列 |
| 使用场景 | 点对点 | 发布/订阅、路由分发 |

## 四、消息可靠性保障

### 防丢失三件套

```
生产者端                    RabbitMQ                    消费者端
    │                          │                          │
    │  ──publish──→            │                          │
    │  (durable exchange)      │                          │
    │                          │  ──deliver──→            │
    │                          │  (durable queue)         │
    │                          │                          │  ──process──→
    │                          │                          │  手动ACK
    │                          │                          │  (autoAck=false)
    │                          │                          │
    │                          │  ←──Nack(requeue=true)──  失败重入队
```

1. **Exchange持久化**（`durable=true`）：RabbitMQ重启后Exchange不丢失
2. **Queue持久化**（`durable=true`）：RabbitMQ重启后Queue不丢失
3. **手动ACK**（`autoAck=false`）：消费者处理完业务才确认

### 失败处理策略

```go
handler := func(msg amqp.Delivery) {
    defer func() {
        if r := recover(); r != nil {
            log.Error("panic recovered:", r)
            msg.Nack(false, true)  // 重入队
        }
    }()
    
    err := processMessage(msg.Body)
    if err != nil {
        log.Error("process failed:", err)
        msg.Nack(false, true)  // 重入队，等待重试
        return
    }
    
    msg.Ack(false)  // 成功，确认消息
}
```

### panic保护

每个消费者都使用 `defer recover()` 防止 panic 导致整个goroutine崩溃。这是MQ消费者的标准实践。

## 五、5个消费队列详解

### 1. custingNft（NFT铸造）

| 项目 | 内容 |
|------|------|
| 触发 | 用户支付成功 |
| 消费内容 | 铸造记录ID |
| 处理逻辑 | 调用Conflux合约Mint → 等待回执 → 上传IPFS → 更新DB |
| 失败后果 | 铸造失败 → 触发退款 |
| 耗时 | 数秒到数分钟（链上确认） |

### 2. shareOrder（分账）

| 项目 | 内容 |
|------|------|
| 触发 | 支付回调成功 |
| 消费内容 | 订单信息 |
| 处理逻辑 | 计算分账比例 → DB事务 → 调用分账API → 完结分账 |
| 失败后果 | 分账失败 → 商户/推广者未收到钱 |
| 耗时 | 数秒（API调用） |

### 3. userCashout（用户提现）

| 项目 | 内容 |
|------|------|
| 触发 | 用户申请提现 |
| 消费内容 | 提现记录ID |
| 处理逻辑 | 调用微信企业付款到零钱 |
| 失败后果 | 用户未收到钱 |
| 耗时 | 数秒 |

### 4. businessCashout（商户提现/进件）

| 项目 | 内容 |
|------|------|
| 触发 | 定时任务（每5分钟） |
| 消费内容 | 商户进件信息 |
| 处理逻辑 | 调用微信商户进件API |
| 失败后果 | 商户无法收款 |
| 耗时 | 数秒 |

### 5. notice（通知）

| 项目 | 内容 |
|------|------|
| 触发 | 各种业务事件 |
| 消费内容 | 通知内容 |
| 处理方式 | **Redis Stream** 消费者组模式（非RabbitMQ） |
| 说明 | 通知允许少量丢失，用Redis Stream替代RabbitMQ |

## 六、Redis Stream 通知消费

```go
// 创建消费者组
XGroupCreateMkStream("notice", "notice_group", "$")

// 阻塞读取
XReadGroup("notice_group", consumerID, "notice", 1, 1*time.Minute)

// 确认消息
XAck("notice", "notice_group", messageId)
```

**为什么通知用Redis Stream而不用RabbitMQ？**
- 通知允许丢失，不需要RabbitMQ的强可靠性
- Redis已经是基础设施，减少组件依赖
- Redis Stream支持消费者组，可以多实例并行消费

## 七、面试常见问题

### Q1: prefetchCount=1 会不会太慢？
> 是的，这是用吞吐量换顺序性。如果不需要严格顺序，可以调大到10-50。我们选择1是因为NFT铸造和分账需要按顺序处理（同一用户的操作）。

### Q2: 消息一直重试失败怎么办？
> 当前设计是无限重入队。生产环境应该加死信队列（DLX），超过重试次数后进入死信队列，人工介入处理。

### Q3: RabbitMQ挂了怎么办？
> 消息会丢失（没有配置消息持久化）。改进方案：开启消息持久化（`DeliveryMode: 2`），但会降低吞吐量。

### Q4: 为什么选RabbitMQ而不是Kafka？
> 我们的场景需要：可靠投递（支付不能丢）、路由灵活（不同消息到不同队列）、中等吞吐量（日几万条）。RabbitMQ更适合。Kafka更适合高吞吐的日志采集场景。
