### 双商户模式

```
┌─────────────────────────────────────────┐
│             UEFUN 平台                   │
│                                          │
│  ┌──────────────┐  ┌──────────────────┐  │
│  │  主商户客户端   │  │  合作伙伴商户客户端 │  │
│  │ (自营商品)     │  │ (服务商/子商户)   │  │
│  │ wxpay/        │  │ partnerwxpay/   │  │
│  └──────┬───────┘  └────────┬─────────┘  │
│         │                   │            │
│         ▼                   ▼            │
│  ┌──────────────────────────────────┐    │
│  │       微信支付 V3 API             │    │

│  └──────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

- **主商户**：平台自营业务（活动报名费、商品购买）
- **合作伙伴商户**：服务商模式（ISV），为入驻商户提供支付能力，涉及 `SubMchid` 子商户号

### 单例模式

```go
type Service struct {
    PrepayService    *jsapi.PrepayService
    RefundService    *refunddomestic.RefundsApiService
}

var (
    instance *Service
    once     sync.Once
)

func GetInstance() *Service {
    once.Do(func() {
        client := internal.GetWechatPayClient()
        if client == nil {
            panic("wechat pay client is nil")
        }
        instance = &Service{
            PrepayService:    jsapi.NewPrepayService(client),
            RefundService:    refunddomestic.NewRefundsApiService(client),
        }
    })
    return instance
}
```

**设计要点**：
- `sync.Once` 保证全局唯一实例（懒加载）
- 启动时 `panic` — fail-fast策略，支付客户端初始化失败应直接终止进程

## 二、支付下单流程

```
用户点击支付
    │
    ▼
生成订单号（商户号前4位 + 时间戳hex + 4字节随机hex，crypto/rand）
    │
    ▼
调用 JSAPI.PrepayWithRequestPayment
    │  参数：AppID, MchID, Description, OutTradeNo, NotifyUrl, Amount, Payer(OpenID)
    │
    ▼
SDK返回 PrepayWithRequestPaymentResponse（含JSAPI调起参数）
    │
    ▼
前端调起微信支付
    │
    ▼
用户输入密码完成支付
    │
    ▼
微信服务器发送支付结果通知 → POST /wxpay_hook
```

### 订单号生成

```go
// 商户号前4位 + 时间戳hex + 4字节随机hex
func GenerateOrderNo(mchID string) string {
    prefix := mchID[:4]
    timestamp := fmt.Sprintf("%x", time.Now().Unix())
    randomBytes := make([]byte, 4)
    crypto/rand.Read(randomBytes)
    random := fmt.Sprintf("%x", randomBytes)
    return prefix + timestamp + random
}
```

**为什么用 crypto/rand 而不是 math/rand？**
- `crypto/rand` 使用操作系统提供的密码学安全随机数
- `math/rand` 是伪随机数，理论上可预测
- 支付订单号需要防伪造和防碰撞

## 三、支付回调与幂等

### 回调处理流程

```
微信POST /wxpay_hook
    │
    ▼
验证签名（SDK自动完成 WithWechatPayAutoAuthCipher）
    │
    ▼
解析通知内容（TransactionId, OutTradeNo, TradeState, Amount）
    │
    ▼
查询订单，校验状态
    │
    ├─ 订单不存在 → 记录日志，返回 SUCCESS（防止重复创建）
    ├─ 订单已支付 → 直接返回 SUCCESS（幂等）
    ├─ 订单已关闭/已退款 → 返回 SUCCESS
    │
    └─ 订单待支付 → 更新为已支付
         │
         ├─ 更新订单状态（DB事务）
         ├─ 发送分账消息到RabbitMQ
         └─ 返回 SUCCESS
```

### 幂等性保障

1. **数据库状态机**：只有"待支付"才能变为"已支付"
2. **回调重复处理**：重复回调时订单已不是"待支付"，直接返回SUCCESS
3. **定时兜底**：`StartUpdateWithdraw` 每小时扫描异常状态的订单

## 四、退款设计

```go
func (s *Service) Refund(orderNo string, totalFee, refundFee int64, reason string) error {
    resp, err := s.RefundService.Create(ctx, &refunddomestic.CreateRequest{
        OutTradeNo:  core.String(orderNo),
        OutRefundNo: core.String(orderNo + "_refund"),
        Amount:      &core.Amount{Total: core.Int64(totalFee), Refund: core.Int64(refundFee)},
        FundsAccount: core.String("REFUND_SOURCE_AVAILABLE"),  // 仅从可用余额退款
    })
}
```

### 退款与分账的互斥逻辑

```
已分账的订单退款前：
1. 先调用 ReturnOrdersApiService 回退分账
2. 分账回退成功后，再调用退款接口
3. 如果分账回退失败，退款不执行
```

**代码中的实现**：分账处理函数 `ShareOrderHandler` 中检查订单分账状态，已分账的订单不允许直接退款。

## 五、分账系统（核心亮点）

### 分账模式

```
用户支付 ¥100
    │
    ├── 冻结在微信（未分账）
    │
    ▼
支付成功 → 发送分账消息到RabbitMQ（shareOrder队列）
    │
    ▼
分账消费者处理：
    │
    ├── 计算分账比例
    │   ├── 净收入 = fee × 90%
    │   ├── 平台服务费 = CNY × 3%
    │   └── 其他收入 = CNY × 7%
    │
    ├── DB事务：创建交易记录 + 更新部落基金 + 保存分账订单
    │
    ├── 调用分账API
    │   ├── 标准分账：CreateOrder（平台商户ID作为接收方）
    │   ├── v1分账：CreateOrderv1（自动完结，Finish=true）
    │   └── v2分账：CreateOrderv2（平台 + 个人推广同时分账）
    │
    └── 成功 → 完结分账（UnfreezeOrder解冻剩余资金）
        失败 → 重入队等待重试
```

### 三种分账接口

| 方法 | 接收方类型 | 完结方式 | 场景 |
|------|-----------|---------|------|
| `CreateOrder` | MERCHANT_ID | 需手动完结 | 通用分账 |
| `CreateOrderv1` | MERCHANT_ID | Finish=true（自动） | 简单场景 |
| `CreateOrderv2` | MERCHANT_ID + PERSONAL_OPENID | 需手动完结 | 含个人推广分账 |

### 特殊子商户处理

```go
if subMchId == "1608905950" {
    // 特定子商户直接完结分账，不解冻
} else {
    // 正常流程：先分账 → 再解冻剩余资金
}
```

### 分账资金生命周期

```
支付成功 → 资金冻结 → 分账 → [回退分账] → 完结分账（解冻剩余）
                                      ↑
                              退款时先回退分账
```

```

    │
```

## 七、面试常见问题

### Q1: 支付回调如果一直收不到怎么办？
> 有定时兜底：`StartUpdateWithdraw` 每小时扫描"支付中"超过一定时间的订单，主动调用微信查单接口确认状态。

### Q2: 分账和退款的互斥逻辑具体怎么实现？
> 分账前检查订单状态，已退款的订单不分账；退款前检查分账状态，已分账的订单先调 `ReturnOrdersApiService` 回退分账，再执行退款。

### Q3: 为什么订单号用crypto/rand？
> 支付订单号需要防碰撞和防伪造。math/rand是伪随机数，种子相同则序列相同；crypto/rand使用操作系统的密码学安全随机源。

### Q4: 服务商模式和直连商户模式有什么区别？
> 直连商户：平台直接收款，分账给商户。
> 服务商模式：平台作为ISV，用子商户号(SubMchid)代收款，微信支付记录中显示的是子商户信息。需要额外的商户进件流程。

### Q5: 分账失败怎么处理？
> 消费者使用 `Nack(false, true)` 重入队，RabbitMQ会重新投递。同时有定时任务扫描长时间"分账中"的订单进行补偿处理。
