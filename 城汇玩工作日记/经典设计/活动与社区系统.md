## 一、活动系统

### 业务模型

```
活动（Activity）
    ├── 基本信息：标题、描述、封面、时间、地点
    ├── 票型（Ticket）：多种票型，每种有独立价格和库存
    ├── 报名（Registration）：用户→票型→订单→支付
    ├── 签到（Sign-in）：活动开始后，组织者确认到场
    ├── 问卷（Questionnaire）：活动后收集反馈
    └── 统计（Stats）：报名数、签到率、收入统计
```

### 核心接口（~40+个）

| 类别 | 接口 | 说明 |
|------|------|------|
| 公开 | GET /activity/list | 活动列表（搜索、筛选、分页） |
| 公开 | GET /activity/detail | 活动详情 |
| 公开 | GET /activity/types | 活动类型列表 |
| 认证 | POST /auth/activity/create | 创建活动 |
| 认证 | PUT /auth/activity/update | 编辑活动 |
| 认证 | POST /auth/activity/register | 报名（创建订单） |
| 认证 | POST /auth/activity/sign | 签到 |
| 认证 | POST /auth/activity/collect | 收藏 |
| 认证 | GET /auth/questionnaire | 问卷 |

### 活动库存扣减（并发安全）

```
用户报名
    │
    ▼
查询票型库存
    │
    ├── 库存不足 → 返回错误
    │
    └── 库存充足
         │
         ▼
    开启DB事务
         │
         ├── SELECT ... FOR UPDATE（行锁，锁定该票型行）
         │
         ├── 二次检查库存（防止幻读）
         │
         ├── UPDATE ticket SET stock = stock - 1, version = version + 1
         │   WHERE id = ? AND version = ?（乐观锁）
         │
         ├── 如果影响行数 = 0 → 库存已被其他请求扣减，返回"库存不足"
         │
         ├── 创建订单记录
         │
         └── 提交事务
```

### 支付超时释放库存

```
定时任务 StartEndActivityFinishOrder（每5小时）
    │
    ▼
扫描"支付中"超过15分钟的订单
    │
    ▼
自动关闭订单 + 释放库存
```

### 活动签到自动标记

```go
// 活动结束后1小时自动标记为已签到
if activity.EndTime.Add(1 * time.Hour).Before(time.Now()) {
    order.IsSign = 1
}
```

## 二、部落社区系统

### 业务模型

```
部落（Tribe）
    ├── 基本信息：名称、描述、封面、类型
    ├── 成员（Member）：加入/退出/踢出
    ├── 管理员（Admin）：创建者 + 指定管理员
    ├── 任务（Task）：发布→申请→审核→完成→发证书
    ├── 商品（Goods）：上架→购买→签收
    ├── 积分（Mark）：获取→消耗→排行榜→月度结算
    ├── 数字藏品（NFT）：铸造→领取→转让
    └── 基金（Fund）：分账收入，用于部落运营
```

### 核心接口（~80+个）

| 模块 | 接口数 | 说明 |
|------|--------|------|
| 部落基础 | ~15 | 创建、编辑、列表、详情、审核、隐藏/显示 |
| 成员管理 | ~10 | 加入、退出、踢出、列表、邀请 |
| 任务系统 | ~15 | CRUD、申请、审核、完成、证书 |
| 商品系统 | ~15 | 上架、购买、签收、收藏、结算 |
| 提现 | ~10 | 申请、审核、记录、银行信息 |
| 数字藏品 | ~15 | 铸造、领取、转让、列表 |

### 部落任务系统

```
发布任务
    │
    ▼
用户申请
    │
    ▼
审核（自动/人工）
    │
    ▼
完成后发放证书
    │
    ▼
发放积分（Mark）
```

### 积分（Mark）体系

```
积分获取：
    ├── 完成任务 → +X Mark
    ├── 每日签到 → +1 Mark
    └── 活动参与 → +X Mark

积分消耗：
    ├── 商品兑换
    └── （其他消费场景）

排行榜：
    ├── 用户个人排名
    └── 部落排名

月度结算：
    ├── 快照当前排名 → MarkHistoricalRanking
    ├── 积分衰减：Mark × 0.4（保留40%）
    └── 赛季概念："第X赛季"（按月份）
```

### 积分月度结算（并发设计）

```go
func StartMarkSettlement() {
    var wg sync.WaitGroup
    wg.Add(2)
    
    // 并行更新用户排名和部落排名
    go func() {
        defer wg.Done()
        updateUserRankings(projectName)
    }()
    go func() {
        defer wg.Done()
        updateTribeRankings(projectName)
    }()
    
    wg.Wait()
}
```

**双协程并行**：用户排名和部落排名互不依赖，并行处理提高效率。

### 部落商品结算

### JWT认证中间件

```go
// router/middlewares/loginCheck/loginCheck.go
func LoginCheck() gin.HandlerFunc {
    return func(c *gin.Context) {
        token := c.GetHeader("Access-Token")
        
        // 解析JWT
        claims, err := jwt.Parse(token, func(token *jwt.Token) (interface{}, error) {
            return []byte(config.App.JwtSecret), nil
        })
        
        // 验证用户存在
        var user model.User
        db.First(&user, claims.UserID)
        if user.ID == 0 {
            c.AbortWithStatusJSON(401, gin.H{"error": "unauthorized"})
            return
        }
        
        // 注入用户信息到context
        c.Set("uid", user.ID)
        c.Next()
    }
}
```

## 五、面试常见问题

### Q1: 活动库存扣减，如果Redis挂了怎么办？
> 我们没用Redis做库存扣减，直接用MySQL行锁+乐观锁双重保障。即使Redis挂了，库存扣减仍然安全。Redis只用于缓存和通知。

### Q2: 为什么签到要自动标记？
> 有些线下活动组织者忘记手动签到，活动结束后1小时自动标记避免数据不完整。

### Q3: 积分衰减（×0.4）的目的是什么？
> 防止老用户积分无限累积导致新用户无法追赶。每个"赛季"（月）重置时保留40%，既奖励活跃用户又保持竞争性。

### Q4: 部落任务审核有自动和人工两种，怎么区分？
> 金额或条件简单的任务自动审核通过（`StartAudit`定时任务），复杂的任务需要管理员手动审核。
